面向对象 & 面向过程

面向对象

  • 优点
    • 易维护、易复用、易扩展，由于面向对象有封装、继承、多态性的特性，可以设计出低耦合的系统，使系统 更加灵活、更加易于维护
  • 缺点
    • 性能比面向过程低

面向过程

  • 优点
    • 性能比面向对象高，因为类调用时需要实例化，开销比较大，比较消耗资源;比如单片机、嵌入式开发、 Linux/Unix等一般采用面向过程开发，性能是最重要的因素。
  • 缺点
    • 没有面向对象易维护、易复用、易扩展

例子(网)

  • 面向过程
    • 蛋炒饭
    • 饭菜交融，味道香美，但换菜只有重做了。
  • 面向对象
    • 盖浇饭
    • 饭菜分离，耦合度低，想换菜很方便，但味道可能没有蛋炒饭号。

特点

封装

  • 隐藏对象的属性和实现细节，仅对外提供公共访问方式
  • private
    • 权限修饰符，修饰成员变量和成员方法
    • private仅仅是封装的一种体现形式，不能说封装就是私有。

继承

  • 好处
    • 提高代码的复用性
    • 提高代码的维护性
    • 让类与类产生关系，是多态的前提
  • 弊端
    • 类的耦合性增强了
  • 特点
    • 子类会继承父类构造函数之外的所有东西
    • private的只是无法访问。
    • 子类重写父类方法时，访问权限不能更低
      • 最好就一致
    • 父类静态方法，子类也必须通过静态方法进行重写
      • 其实这个算不上方法重写，但是现象确实如此(静态只能覆盖静态)
    • 如果子父类有相同的成员变量
      • 在堆内存的对象空间中会分别有存子/父成员变量的地方
      • 实际基本不会出现这种情况。
  • 开发原则：高内聚，低耦合
    • 耦合：类与类的关系
    • 内聚：自己完成某件事的能力

继承中 成员变量的访问

直接通过子类对象访问成员变量

  • 等号左边是谁，就优先使用谁，没有则往上找

间接通过成员方法访问成员变量

  • 该方法属于谁，就优先用谁，没有则往上找。

继承中 成员方法的访问

  • 创建的对象是谁，就优先使用谁，没有则往上找

注

  • 无论是成员变量还是方法，都只会往上找，父类不会找子类。
    • 父不知道谁是自己儿子，儿子直到谁是自己父。

多态

  • 前提
    • 继承关系
    • 方法重写
    • 父类引用指向子类对象。
  • 编译运行(访问特点)
    • 成员变量
      • 编译运行都看左边
    • 一般成员函数
      • 编译看左边，运行看右边
      • 动态绑定
    • 静态成员函数()
      • 编译运行都看左边
  • 优点
    • 提高代码维护性(继承保证)
    • 提高代码扩展性(多态保证)
  • 缺点
    • 不能使用子类的特有属性和方法
  • instanceof
    • 判断是否是某种类型
      • boolean b = a instanceof Object
      • 只能用来判断引用数据类型？

(抽象)

  • abstract
    • 修饰类，方法
    • 不能修饰成员变量，没有意义。
  • abstract不共存的关键字
    • static
      • 类名调用抽象方法没有意义
    • private
      • abstract是为了重写
      • private修饰的虽然可以继承，但是无法访问，也就无法重写
    • final
      • abstract是为了重写，final修饰不能重写

  • protected
    • 与基类不在同一个包中的子类，只能访问自身从基类继承而来的受保护成员，而不能访问基类实例本身的受保护成员。
      • 安
        • 在不同包的子类中
          • 可以访问子类本身继承过来的protected成员
          • 不能访问父类实例本身或者其它子类实例继承的protected成员
          • 说白了就是只能用自己的实例调用protected成员，不能用父类实例或者父类的其它子类的实例调用protected成员
        • 在同包的子类中
          • 可以访问子类本身和父类的protected成员
          • 不能访问父类其它子类的protected成员
      • 例 Object类的clone()是protected
        • 在an.test.A中
          • this.clone() //正确
          • new A().clone() //正确
          • new Object().clone() //错误
          • new B().clone() //错误
  • this
    • 代表当前对象的引用,谁来调用我,我就代表谁。this代表的是调用对象
    • 编译时代表当前类。
    • 运行时代表当前运行类。
      • 运行时的类中this和super代表自己和父类。
      • 即使是父类的方法中有this也代表的当前运行类。
      • this.字段
        • 就是当前类的字段
      • this.方法
        • 指当前运行类的方法
  • super
    • 代表当前对象父类的引用
  • static
    • 随着类的加载而加载
    • 优先于对象存在
    • 被类的所有对象共享
    • 可以通过类名调用
    • 在静态方法中是没有this关键字的
    • 静态方法只能访问静态的成员变量和静态的成员方法
    • 静态变量存储于方法区的静态区
  • final
    • 修饰类，类不能被继承
    • 修饰变量，变量就变成了常量，只能被赋值一次
      • 基本类型，是值不能被改变
      • 引用类型，是地址值不能被改变,对象中的属性可以改变
    • 修饰方法，方法不能被重写
    • final修饰变量的初始化时机
      • 显示初始化
      • 在对象构造完毕前即可

内部类

概述

  • 内部类可以直接访问外部类的成员，包括私有。
    • 内部类持有一个外部类的引用
    • 格式：外部类名.this
  • 外部类要访问内部类的成员，必须创建对象。
  • 内部类格式
    • 内部类
      • 外部类名.内部类名 变量名=外部类对象.内部类对象
      • Out.Inner in=new Out().new Inner();
    • 静态内部类
      • 外部类名.内部类名 变量名=外部类名.内部类对象
      • Out.Inner in=new Out.Inner();
        • 这个new其实是针对Inner的，习惯写在前边
        • Out.new Inner()

局部内部类

  • 内部类可以放在局部（如方法中）
    • 局部内部类只能在其所在的方法中访问
  • 局部内部类在访问他所在方法中的局部变量必须用final修饰,为什么?
    • 因为当调用这个方法时,局部变量如果没有用final修饰,他的生命周期和方法的生命周期是一样的,当方法弹栈,这个局部变量也会消失,那么如果局部内部类对象还没有马上消失想用这个局部变量,就没有了,如果用final修饰会在类加载的时候进入常量池,即使方法弹栈,常量池的常量还在,也可以继续使用
    • 但是jdk1.8不加final也不报错。但实际还是final的。

接口

  • 成员变量都是常量
    • public static final
      • 默认
  • 成员方法
    • public abstract
      • 默认
  • 接口中没有构造方法
  • 接口不能有静态方法
    • 接口直接调用静态抽象方法没意义

抽象类

  • 抽象类有构造方法
  • 可以有抽象方法也可以有具体方法
  • 有抽象方法必须是抽象类
  • 抽象类不必须有抽象方法
  • 抽象类可以有静态方法

对象实例化过程

  1. 加载class文件
    • 有父类先加载父类
  2. 静态代码块
    • 静态代码块只会执行一次
  3. 在堆内存开辟空间分配地址
  4. 在堆内存的对象空间中对属性进行默认初始化
  5. 调用构造函数
    • 第一行会先调用父类构造函数
  6. 对属性进行显示初始化
  7. 构造代码块
    • 每次创建对象都会执行构造代码块
  8. 执行构造函数剩余部分。
