Web容器

  • 所有的程序运行都需要有一个必要的运行环境。这个环境可以是软件，也可以是硬件，或者是软件和硬件的结合。比如说Windows操作系统需要运行在硬件基础上；Office软件需要运行在操作系统上。并且程序与运行环境之间会有一定的数据交换，比如操作系统会将运行指令传递给硬件，硬件也会将指令运行结果传递给操作系统
  • Java Web程序也需要一个运行环境才能够执行。这种运行Java Web程序的环境被称为Web容器，Java Web程序与Web容器之间存在数据交互。

独立的Java Web容器

  • 在这种容器里面只能运行Web程序，这种容器一般也叫做Web服务器，如Tomcat

与其他Java EE容器混合在一起的Web容器

  • Web容器负责运行Web程序，其他容器负责运行EJB等程序，如WebLogic

web容器与web应用间的数据交互

  • web容器接收客户端请求封装HttpServletRequest对象，并传递给web应用
  • web应用处理请求
  • web容器将结果封装到HttpServletResponse对象，响应给客户端
  • 注
    • 这两个对象仅存在于Web容器和Web应用当中，与浏览器没有关系

tomcat和servlet的区别

  • Tomcat 是Web应用服务器,是一个Servlet/JSP容器
  • servlet是一种运行在服务器端的Java应用程序

##CATALINA_HOME和CATALINA_BASE区别

CATALINA_HOME

  • 是Tomcat的安装目录
  • 类似于JAVA_HOME环境变量
  • 如果配置了就能在任意路径执行bin目录下的bat或者exe
  • 通常不配置

CATALINA_BASE

  • 是Tomcat的工作目录
  • 如果CATALINA_BASE没有值的话，它默认的值就是CATALINA_HOME

tomcat目录

  • 其中只有 bin 和 lib 目录被多个tomcat示例公用
  • 其它目录conf、logs、temp、webapps和work 每个Tomcat实例必须拥有其自己独立的备份
  • catalina.home(安装目录)：
    • 指向公用信息的位置，就是bin和lib的父目录
  • catalina.base(工作目录)：
    • 指向每个Tomcat目录私有信息的位置，就是conf、logs、temp、webapps和work的父目录。
  • 仅运行一个Tomcat实例时，这两个属性指向的位置是相同的

  • CATALINA_HOME代表了一个tomcat，需要tomcat的bin,lib...

  • 小量数据载体，且只更新在web页面中的一个地方

优点

缺点

  • XML文件格式相当严格，出错后，responseXML属性返回NULL，如果XML文件过长，导致解析效率低下

适合

  • 大量具有层次数据载体
  • response.setContentType("text/xml;charset=utf-8")

  • JSON是不能完完全全替代XML的，只能在定义对象和数据交换语言方面替代

xd-Ajax

发表于 2018-12-11 | 更新于 2019-03-08 | 分类于 UI

概述

  • Asynchronous JavaScript And XML
  • 传统的网页（不使用 AJAX）如果需要更新内容，必需重载整个网页面。
  • 通过在后台与服务器进行少量数据交换，AJAX 可以使网页实现异步更新。这意味着可以在不重新加载整个网页的情况下，对网页的某部分进行更新
  • 传统
    • 客户端发送请求到服务端，服务端返回数据给客户端。
  • Ajax
    • 客户端通过javascript call ajax引擎，由ajax engine发送请求到服务端，服务端，返回数据给ajax engine再由ajax engine把数据交给客户端

原理

  • 通过XmlHttpRequest对象来向服务器发异步请求，从服务器获得数据，然后用javascript来操作DOM而更新页面。

同步&异步

  • 客户端与服务器相互通信时
    • 同步
      • 客户端必须等待服务器的响应，等待期间客户端不能做其它操作
    • 异步
      • 客户端不需要等待服务器的响应，在服务器处理请求的过程中，客户端可以进行其它操作

XMLHttpRequest

  • 异步通讯对象
  • XMLHttpRequest 是 AJAX 的基础。
  • IE5 和 IE6 使用 ActiveXObject

事件

  • onreadystatechange
  • 存储函数（或函数名），每当 readyState 属性改变时，就会调用该函数。

属性

  • readyState
    • 存有 XMLHttpRequest 的状态。从 0 到 4 发生变化。
    • 0
      • 请求未初始化
    • 1
      • 服务器连接已建立
      • 已经调用了open()但没有真正发送请求
    • 2
      • 请求已接收
      • 已调用send()但暂时没有到达服务器
    • 3
      • 请求处理中
    • 4
      • 请求已完成，且响应已就绪
  • status
    • 200
      • OK
    • 404
      • 未找到页面
  • responseText
    • 获得字符串形式的响应数据。
  • responseXML
    • 获得 XML 形式的响应数据。

方法

  • open(method,url,async)
    • method
      • 请求的类型；GET 或 POST
    • url
      • 文件在服务器上的位置
    • async
      • true（异步，默认）
      • false（同步）
  • send(string)
    • string
      • 发送的数据，仅用于 POST 请求。
      • POST发送前要设置请求头。
        • setRequestHeader("content-type","application/x-www-form-urlencoded")

概述

  • 基于对象和事件驱动的解释性脚本语言。
    • 基于对象
      • JS本身就有一些现成的对象可供程序员使用，也可自定义对象。
    • 事件驱动
      • JS代码写好后，需要外界触发后，方可运行
      • 例如：单击事件，定时执行。。。
    • 解释性
      • 每次运行JS代码时，得需要将原代码一行一行的解释执行
      • 相对编译型语言（例如：Java、C++）执行速度相对较慢
    • 基于浏览器的动态交互网页技术
      • JS嵌入到HTML中，可以不需要服务器支持，直接由浏览器解释执行
      • JS嵌入到JSP或Servlet中，必需要服务器支持，直接由浏览器解释执行

特点

  • 交互性
  • 安全性
    • js不能访问本地磁盘
  • 跨平台

  • 面向对象

    • 一个工程师再图纸上设计出一栋房子的样子，然后其它工程师按照这个图纸的设计去建造房子

优点

  • 客户端不需要负责对象的创建，从而明确了各个类的职责

缺点

  • 静态工厂类负责所有对象的创建，如果有新的对象增加，或者某些对象的创建方式不同，就要不断的修改工厂类，不利于后期维护。

概述

  • 抽象工厂类负责定义创建对象的接口，具体对象的创建由继承抽象工厂的具体类实现。
  • 其实就是创建一个抽象的顶级接口，然后具体对象有一个针对的工厂类实现顶级接口。新增对象时，新增一个对象工厂即可，不必修改其它工厂。

适配器

概述

  • 假如有个接口有许多方法，我们要实现接口，就要覆写所有的方法，有些方法我们根本用不到。
  • 现在有个类或抽象类，实现接口，把所有方法或不常用的方法覆写。可能方法内部什么也没做，就是简单的覆写。
  • 这样我们继承这个类或抽象类。只需要覆写想要覆写的方法或者实现没有覆写的抽象方法即可。

例

  • Servlet接口 GenericServlet抽象类
  • Servlet接口中有五个方法，如果我们实现了Servlet接口，就必须要覆写这五个方法，有的方法，我们根本用不到。
  • 现在有了GenericServlet抽象类实现了Servlet接口，我们继承GenericServlet即可。

代理模式

  • Proxy Pattern

结构

  • 目标对象
    • 真正要访问的对象
  • 代理对象
  • 代理类
    • 用来生成代理对象
  • 目标对象和代理对象实现同一接口

原理(安)

  • 目标对象和代理对象有着相同的功能。
  • 做一件事，内部其实还是目标对象在做，只是表面是代理对象在执行行为。
  • 代理对象的行为内部在目标对象做的事前后还可以自己做一些事情。
  • 例
    • 潘金莲和王婆有相同功能（实现同一个接口）抛媚眼
    • 潘金莲（目标对象）给西门庆抛媚眼（事情），不直接这么干，表面上由王婆（代理对象）抛媚眼，其实真正抛媚眼的还是潘金莲。
    • 在王婆的抛媚眼方法内部，执行的是潘金莲的抛媚眼，而在潘金莲抛媚眼前后，王婆还可以做一些事情(如之前拿些好处，之后干掉武大郎)

动态代理

概述

  • 动态代理可以直接给某一个目标对象生成一个代理对象，而不需要代理类存在。
  • 目标对象和代理对象实现同一接口是为了保证方法一致
  • 动态代理与代理模式原理是一样的，只是它没有具体的代理类，直接通过反射生成了一个代理对象。

基于接口 & 基于继承

  • 基于接口，接口有的方法才能代理，目标类中特有的不能代理
  • 基于继承，由于代理类继承了目标类，目标类中所有方法都能代理。

        • args

    • 我们要执行一个目标类A的目标方法add()
      • 原
        • A a=new A()
        • a.add()
      • 使用代理
        • A $a=Proxy.newProxyInstance(classLoader,interfaces,h)
          • 获取A的代理对象$a
        • $a.add()

        • invoke()方法中的method.invoke(a，实际参数)
          • 执行的是目标类的方法。即a.add()
          • 这样就通过代理完成了目标类A的目标方法add()
        • 而在invoke()方法中调用method.invoke(a，实际参数)前后还可以做一些其它事情。
          • 这样通过代理我们不仅可以实现原来调用目标方法的目的，还可以在目标方法执行前后做一些事。

概述
  • 没有接口,只有实现类。
  • 采用字节码增强框架 cglib，在运行时 创建目标类的子类，从而对目标类进行增强。
  • 可以对任何类生成代理，原理是对目标对象进行继承代理
